# AGNES Node-HUB Message Schema v0.1

## 1) Objective

Define a minimal, stable, and traceable contract for message exchange between:

- contact node (XIAO ESP32-C3)
- environmental node (ESP32-S3)
- HUB (ESP32-S3)

This schema is designed to:

- integrate early sensor hello-world payloads without structural rework;
- standardize collection, validation, and logging;
- provide consistent inputs for the Veto Engine.

---

## 2) Protocol principles

- MVP simplicity.
- Deterministic and auditable behavior.
- Tolerance to packet loss.
- Extensibility without breaking compatibility.

---

## 3) Transport model (MVP)

- Primary transport: ESP-NOW.
- Payload format: compact UTF-8 JSON.
- One message object per packet.
- Version control: mandatory `schema_version` field.

> Note: if payload exceeds ESP-NOW limits, send derived summaries from node logic and keep raw high-rate data local for debug channels.

---

## 4) Common envelope (mandatory)

All message types must include:

- `schema_version` (string), e.g. `"0.1"`
- `msg_type` (string), e.g. `"heartbeat"`, `"evidence"`
- `msg_id` (string), UUID or monotonic unique node ID
- `ts_ms` (number), epoch in milliseconds
- `node_id` (string), e.g. `"contact_node"`, `"env_node"`, `"hub"`
- `seq` (number), per-node sequence counter

Envelope example:

```json
{
  "schema_version": "0.1",
  "msg_type": "heartbeat",
  "msg_id": "contact_node-1700000000123-1024",
  "ts_ms": 1700000000123,
  "node_id": "contact_node",
  "seq": 1024
}
```

---

## 5) Message types

### 5.1 HEARTBEAT

Purpose: node health and link availability.

Additional fields:

- `uptime_ms` (number)
- `rssi_dbm` (number)
- `battery_pct` (number, optional)
- `firmware_version` (string)
- `sensor_status` (object)

Example:

```json
{
  "schema_version": "0.1",
  "msg_type": "heartbeat",
  "msg_id": "env_node-1700000001123-2048",
  "ts_ms": 1700000001123,
  "node_id": "env_node",
  "seq": 2048,
  "uptime_ms": 530001,
  "rssi_dbm": -62,
  "firmware_version": "env-0.1.0",
  "sensor_status": {
    "mmwave": "ok",
    "thermal": "ok"
  }
}
```

### 5.2 EVIDENCE

Purpose: processed data from local sensing or local fusion logic.

Additional fields:

- `sensor_type` (`ecg`, `ppg`, `imu`, `mmwave`, `thermal`, `fusion`)
- `signal_type` (`electrical`, `optical`, `mechanical`, `thermal`)
- `value` (number or object)
- `quality_score` (`0.0-1.0`)
- `confidence_score` (`0.0-1.0`)
- `context_tags` (array of strings)

PPG/ECG example:

```json
{
  "schema_version": "0.1",
  "msg_type": "evidence",
  "msg_id": "contact_node-1700000002123-2049",
  "ts_ms": 1700000002123,
  "node_id": "contact_node",
  "seq": 2049,
  "sensor_type": "ppg",
  "signal_type": "optical",
  "value": {
    "hr_bpm": 132,
    "spo2_pct": 96
  },
  "quality_score": 0.89,
  "confidence_score": 0.84,
  "context_tags": ["stable_rest"]
}
```

Motion evidence example:

```json
{
  "schema_version": "0.1",
  "msg_type": "evidence",
  "msg_id": "env_node-1700000003123-2050",
  "ts_ms": 1700000003123,
  "node_id": "env_node",
  "seq": 2050,
  "sensor_type": "mmwave",
  "signal_type": "mechanical",
  "value": {
    "resp_rate_bpm": 28,
    "motion_index": 42
  },
  "quality_score": 0.91,
  "confidence_score": 0.86,
  "context_tags": ["resp_detected", "movement_moderate"]
}
```

Thermal zonal example:

```json
{
  "schema_version": "0.1",
  "msg_type": "evidence",
  "msg_id": "env_node-1700000060123-2060",
  "ts_ms": 1700000060123,
  "node_id": "env_node",
  "seq": 2060,
  "sensor_type": "thermal",
  "signal_type": "thermal",
  "value": {
    "zones_c": {
      "head": 35.2,
      "upper_limbs_proximal": 34.1,
      "upper_limbs_distal": 32.6,
      "thorax": 36.0,
      "abdomen": 35.8,
      "lower_limbs_proximal": 33.7,
      "lower_limbs_distal": 31.9
    },
    "core_periphery_delta_c": 3.4,
    "thermal_trend": "falling"
  },
  "quality_score": 0.78,
  "confidence_score": 0.74,
  "context_tags": ["thermal_update_60s"]
}
```

### 5.3 EVENT_CANDIDATE

Purpose: optional in early MVP; typically generated by HUB.

Additional fields:

- `candidate_type`
- `candidate_score`
- `supporting_evidence_ids` (array)
- `conflicting_evidence_ids` (array)
- `temporal_window_ms`

### 5.4 COMMAND

Purpose: control commands from HUB to nodes.

Additional fields:

- `command` (e.g. `set_rate`, `self_test`, `sync_time`)
- `params` (object)

### 5.5 ACK

Purpose: command receipt acknowledgment.

Additional fields:

- `ack_for_msg_id`
- `status` (`ok` or `error`)
- `error_code` (optional)

### 5.6 ERROR

Purpose: fault notification and diagnosability.

Additional fields:

- `error_class` (`sensor`, `transport`, `parser`, `runtime`)
- `error_code`
- `error_text`
- `related_msg_id` (optional)

---

## 6) Ordering, deduplication, and replay safety

- Use `(node_id, seq)` for ordering and duplicate detection.
- Ignore packets with stale sequence values.
- Preserve all accepted messages in append-only order on HUB logs.
- Optionally include `boot_id` to detect node restarts cleanly.

---

## 7) Minimum quality and confidence policy (MVP)

- `quality_score < 0.40`: evidence cannot support promotion.
- `confidence_score < 0.50`: evidence is contextual only.
- Critical promotion requires at least two independent supporting evidences above quality/confidence thresholds.

---

## 8) Backward compatibility

- `schema_version` required in every message.
- New optional fields must not break old parsers.
- Breaking changes require explicit version bump and migration notes.

---

## 9) Security and integrity (MVP baseline)

- Validate payload shape before processing.
- Cap max payload size to prevent parser overload.
- Reject malformed JSON and log parser errors.
- Never trust node-level timestamps without envelope consistency checks.

---

## 10) Validation checklist

- Envelope fields present in 100% of messages.
- Sequence monotonicity per node.
- HEARTBEAT periodicity stable.
- EVIDENCE payloads parse correctly for all supported sensor types.
- COMMAND/ACK round-trip traceable by `msg_id`.
- ERROR records include actionable diagnostics.
